## Evaluation Issues

### Hybrid mods not supported
Hybrid mods (e.g., "Increased Physical Damage + to Accuracy Rating" on a staff) are treated as separate one-line mods during parsing, but they share the same prefix/suffix slot. This means:
- Open mod count may be wrong (counting one hybrid as two mods)
- StatValue matching works (finds the stat line) but mod-level predicates like `ModCount` or `ModTier` may double-count
- poe-item's resolver needs to merge stat lines that belong to the same hybrid mod group

**Action**: Investigate how Ctrl+Alt+C text represents hybrid mods. Add fixtures, update poe-item resolver to group them correctly.

### Tier/Rank color scale needs rework
Current tier coloring uses a flat 4-level system (T1 = best, T2-3, T4-5, low). Problems:

1. **Mod-relative scaling**: Each mod has its own tier range. A mod with only T1-T3 means T3 is the worst, not "mid". The color should reflect position within that mod's range, not absolute tier number.
2. ~~**Rank vs Tier**: Crafted mods and some influence mods use Rank instead of Tier, where higher = better (Rank 3 > Rank 1). Currently colored incorrectly.~~ (FIXED — `classify_rank()` in poe-data handles reversed direction)
3. **Data source**: poe-data should own a helper that returns a normalized quality level given a mod's tier/rank and the full tier range for that mod family. App just maps quality → color.

**Action**: Add tier range data to poe-data's `domain.rs` or compute from Mods table. Expose `tier_quality(mod_name, tier) -> Quality` that accounts for tier count and rank direction.

## Future Features

### Pseudo (sum) stats
Like poe.trade's pseudo mods — `pseudo total life > 140` sums all matching stat lines across mods. New predicate type `PseudoStatValue` in poe-eval. The term "pseudo" is established PoE community convention.

### Item browser for rule building
When building rules (e.g., for shields), users need to look up base types, mods, etc. Currently requires googling. We have all this data in poe-data — could render a browsable item/base type database with PoE assets. Useful for Base Type field in rules ("what's the best armour shield base?").

### HasStatText deprecation
`HasStatText` predicate uses text `contains` matching. Should be deprecated in favor of `HasStatId` (already exists). No new rules should use text matching.
